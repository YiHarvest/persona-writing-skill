# 文风改写主流程

## 阶段 0：识别模式

| 情况 | 人物卡来源 | 流程 |
|---|---|---|
| 用户明确指定卡 | 指定卡 | 立骨 → 画皮 → 动嘴 |
| `current --json` 返回有效当前卡 | 当前卡 | 立骨 → 画皮 → 动嘴 |
| 无指定、无当前卡 | 动态匹配 | 立骨 → 铸魂 → 画皮 → 动嘴 |
| 只要求匹配或创建卡 | 卡片系统 | 画像 → 匹配 → 确认或生成 → 保存 |

指定卡名称不明确时，使用 `list --json` 的结果做精确或唯一近似匹配；出现多个候选时才询问。`current --json` 未设置或指针失效时视为没有当前卡。

## 阶段 1：准备

1. 提取原文、载体、受众、目标、长度和用户边界。
2. 用户未说明载体时，根据文本形态合理推断，不因非关键缺失中断。
3. 保留本轮原文和版本上下文，供反馈后立即重写。

## 阶段 2：选卡与改写

1. 按 `10-立骨.md` 完整拆解原文。
2. 需要自动选卡时按 `11-铸魂.md` 动态匹配。
3. 按 `12-画皮.md` 生成初稿。
4. 按 `13-动嘴.md` 终检。
5. 按 `15-最终输出规范.md` 输出。

## 阶段 3：反馈闭环

把用户不满意分为两类：

### A. 执行问题

信号：太长、开头夸张、节奏慢、数据少、某段太激烈、标签不对等。

处理：保留当前人物卡，把反馈转成明确参数，重新执行画皮和动嘴。记录有长期价值的偏好。

### B. 人物卡问题

信号：不像我、领域不对、受众不对、价值立场不对、整体气质不对。

处理：重新提取画像和不匹配点，生成专属卡，完成质量检查；在用户已要求保存的情况下调用 `save --set-current`，然后自动重写刚才原文。

不确定属于哪类时，先判断反馈能否通过 1–2 个局部参数解决；能则按执行问题处理，避免不必要地新建卡。

## 异常处理

- 指定卡不存在：列出实际存在的最接近候选；不虚构卡片。
- 原文缺失：请求用户提供原文，这是改写的必要输入。
- 原文事实明显冲突：指出冲突并保守改写，不擅自选一边。
- 专属卡保存失败：仍可在当前对话使用已生成卡片，但明确说明未持久化。
- 某模块无法读取：使用已读取的信息完成安全的最小版本，并说明缺失项。
